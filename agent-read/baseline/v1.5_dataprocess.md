# V1.5 Data Process Audit

## 1. Goal

This note audits the RoboTwin raw data format and the LingBot-VA post-train preprocessing path for one specific question:

- is the pose/action representation consistent from RoboTwin collection to LingBot usage?
- if not, where is the mismatch most likely happening?

The focus here is `place_can_basket` data collected under:

- `/home/zaijia001/ssd/RoboTwin/data/place_can_basket/demo_clean_large_d435`

and converted into the LingBot post-train bundle under:

- `/home/zaijia001/ssd/RoboTwin/data/place_can_basket/lingbot-posttrain-demo_clean`

## 2. Short Verdict

Position ordering looks consistent.

Quaternion ordering does **not** look consistent.

The highest-confidence issue is:

- RoboTwin raw `endpose` is stored as `[x, y, z, qw, qx, qy, qz]`
- LingBot preprocessing and eval helper code treat the quaternion block as if it were `[qx, qy, qz, qw]`

This can directly produce the kind of symptom you described:

- object motion direction looks wrong
- end-effector orientation looks obviously off
- downstream placement or grasp approach angles become unstable or nonsensical

## 3. What RoboTwin Actually Stores

### 3.1 Raw collection source

RoboTwin saves `endpose` in:

- `/home/zaijia001/vam/RoboTwin-lingbot/envs/_base_task.py`

The collection path is:

1. `get_obs()`
2. `self.get_arm_pose("left" / "right")`
3. `self.robot.get_left_ee_pose()` / `self.robot.get_right_ee_pose()`
4. `self.robot._trans_endpose(...)`

The key point is in:

- `/home/zaijia001/vam/RoboTwin-lingbot/envs/robot/robot.py`

`_trans_endpose(...)` returns:

- `position.tolist() + t3d.quaternions.mat2quat(rotation).tolist()`

and `transforms3d.quaternions.mat2quat(...)` uses:

- `wxyz`

So RoboTwin raw `endpose` is effectively:

- `[x, y, z, qw, qx, qy, qz]`

### 3.2 Sample from your real data

From:

- `/home/zaijia001/ssd/RoboTwin/data/place_can_basket/demo_clean_large_d435/data/episode0.hdf5`

the first left-arm pose is approximately:

```text
[-0.2979, -0.3138, 0.9420, 0.7000, -0.0000, -0.0000, 0.7141]
```

If interpreted correctly as `wxyz`, its Euler angle is approximately:

```text
roll  ~= 0 deg
pitch ~= 0 deg
yaw   ~= 91 deg
```

If misread as `xyzw`, the same numeric tuple becomes approximately:

```text
roll  ~= 89 deg
pitch ~= 0 deg
yaw   ~= 0 deg
```

That is not a small numerical drift. It is a fundamentally different orientation.

## 4. What LingBot Preprocessing Does

### 4.1 Conversion script

The post-train converter is:

- `/home/zaijia001/vam/lingbot-va/script/prepare_robotwin_posttrain.py`

It reads:

- `endpose/left_endpose`
- `endpose/right_endpose`
- `endpose/left_gripper`
- `endpose/right_gripper`

and concatenates them directly:

```text
[left_pose, left_gripper, right_pose, right_gripper]
```

No quaternion reordering happens there.

### 4.2 Semantic labels in the converter

The converter assigns these state names:

```text
left_eef_x
left_eef_y
left_eef_z
left_eef_qx
left_eef_qy
left_eef_qz
left_eef_qw
...
```

This means the converter **names** the quaternion as `qx qy qz qw`, but the raw values copied in are still coming from RoboTwin `wxyz`.

So even before training starts, the bundle already has a semantic mismatch:

- numeric storage: `qw qx qy qz`
- declared meaning: `qx qy qz qw`

## 5. What LingBot Training Data Loading Does

The training-side dataset logic is:

- `/home/zaijia001/vam/lingbot-va/wan_va/dataset/lerobot_latent_dataset.py`

The critical function is:

- `get_relative_pose(pose)`

It does:

```python
rot = R.from_quat(pose[:, 3:7])
```

But `scipy.spatial.transform.Rotation.from_quat(...)` expects:

- `xyzw`

So if the raw pose block is actually `wxyz`, this step computes the relative rotation using the wrong convention.

That means the training data pipeline likely introduces a rotation error here:

- raw RoboTwin quaternion is `wxyz`
- dataset code interprets it as `xyzw`

This is a high-confidence inconsistency.

## 6. What Eval Does

The eval client composes predicted relative actions back onto the initial pose in:

- `/home/zaijia001/vam/lingbot-va/evaluation/robotwin/eval_polict_client_openpi.py`

The relevant helper is:

- `add_eef_pose(new_pose, init_pose)`

It also does:

```python
new_pose_R = R.from_quat(new_pose[3:7])
init_pose_R = R.from_quat(init_pose[3:7])
out_rot = (init_pose_R * new_pose_R).as_quat()
```

Again, this assumes:

- `xyzw`

So if the incoming pose convention is actually `wxyz`, eval-time rotation composition is also wrong.

This means the same convention problem likely exists in both:

- training data construction
- eval-time relative-to-absolute action reconstruction

## 7. Position Handling Audit

I did **not** find evidence that `x/y/z` ordering is swapped.

The position path is:

1. RoboTwin stores `[x, y, z, ...]`
2. converter copies those three values directly
3. dataset computes relative translation with:
   - `relative_trans = trans - trans[0:1]`
4. eval reconstructs with:
   - `out_trans = new_pose[:3] + init_pose[:3]`

So the position convention itself appears internally consistent.

Two important notes:

1. The relative translation is in world coordinates, not local gripper coordinates.
2. The saved `endpose` is the RoboTwin EE control pose from `get_left_ee_pose()` / `get_right_ee_pose()`, not the TCP-center pose from `get_left_tcp_pose()` / `get_right_tcp_pose()`.

Those two details may affect behavior or intuition, but they are not the same kind of hard mismatch as the quaternion ordering issue above.

## 8. Action Channel Usage

For RoboTwin config:

- `/home/zaijia001/vam/lingbot-va/wan_va/configs/va_robotwin_cfg.py`

the used action channels are:

```text
left:  0..6 + gripper
right: 7..13 + gripper
```

So the quaternion channels are not dropped.

This matters because it means the ordering issue is not harmless metadata. The model really sees and predicts those rotation dimensions.

## 9. Consistency Table

### 9.1 Confirmed consistent

- raw pose translation order is `x y z`
- converter copies position values without reordering
- action/state 16D layout is preserved across raw HDF5 and bundle
- gripper is stored as normalized scalar in `[0, 1]`

### 9.2 Confirmed inconsistent

- RoboTwin raw quaternion source is `wxyz`
- converter labels it as `qx qy qz qw`
- training loader `get_relative_pose(...)` interprets it with `scipy` as `xyzw`
- eval `add_init_pose(...)` also interprets it with `scipy` as `xyzw`

## 10. Most Likely Root Cause

If your newly collected data looks like “position and orientation are obviously wrong”, the strongest current hypothesis is:

- quaternion convention drift between RoboTwin and LingBot

More concretely:

1. RoboTwin saves `qw qx qy qz`
2. LingBot preprocessing treats those four numbers as if they were `qx qy qz qw`
3. relative rotation targets are therefore built incorrectly
4. eval reconstruction also composes rotations with the same wrong assumption

This is the first thing I would fix before trusting any post-train result on your new dataset.

## 11. Practical Recommendation

I have not changed code in this audit. This file is only the investigation record.

Recommended next step:

1. define one canonical quaternion convention for this project
2. patch the converter, dataset loader, and eval reconstruction to all use the same convention explicitly
3. regenerate one small post-train bundle
4. re-run a tiny smoke train/eval comparison against the current bundle

The safest canonical choice here is probably:

- store and document RoboTwin pose tensors explicitly as `wxyz`

and only convert to `xyzw` at the exact places where `scipy` is called.
